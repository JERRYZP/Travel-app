## 三、技术方案

### 3.1 架构
```
云函数定时触发器（Cron）
    ↓
Playwright + stealth（云函数内运行）
    ↓
动态代理 IP 池（快代理/阿布云）
    ↓
目标官网/预约页面
    ↓
解析 HTML → 提取日期数据
    ↓
写入云数据库（spots 表更新 earliestDate / rules 表更新 lastCheckedDate）
```

### 3.2 反爬策略
1. **特征隐藏**：`playwright-extra-plugin-stealth` 抹除 `webdriver` 标记
2. **指纹随机化**：随机 User-Agent + viewport + locale
3. **行为模拟**：页面加载后随机延迟 2-5 秒，模拟鼠标滚动
4. **代理 IP**：仅在高频时段（放票前10分钟）启用动态代理，日常低频抓取用云函数默认出口 IP
5. **分时段频率**：
   - 日常（00:00-放票前1小时）：30分钟/次
   - 放票前1小时-放票后10分钟：10秒/次（仅P0景点）
   - 深夜（01:00-05:00）：停止抓取

### 3.3 异常处理
- 页面改版/结构变化 → 抓取失败 → 云函数自动发送告警（企业微信/邮件）
- IP 被封 → 自动切换代理 IP，连续失败3次后暂停该景点抓取并告警
- 验证码拦截 → 标记为"需人工介入"，跳过该次抓取

## 六、风险与兜底

| 风险 | 概率 | 影响 | 应对方案 |
|------|------|------|---------|
| 官网改版导致抓取失效 | 中 | 高 | 监控心跳 + 一键报错机制 + 快速修复（AI辅助重写选择器） |
| 官方加强反爬（封IP/验证码） | 中 | 高 | 代理IP池 + 降低频率 + 众包数据补充 |
| 小程序闭环数据无法网页抓取 | 高 | 中 | V1优先做有网页版的；V2引入RPA |
| 法律风险 | 低 | 高 | 仅抓取公开信息，不涉及代抢，显著位置声明"仅供参考" |
