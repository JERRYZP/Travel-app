# 景点预约提醒小程序：产品需求与技术方案说明书

## 1. 产品背景与核心价值

**定位**：极致的抢票信息聚合与提醒工具

**业务逻辑**：纯粹的实时信息同步与闹钟式提醒。用户收到提醒后，需手动跳转至官方渠道抢票，不涉及代抢业务

**目标用户**：自由行游客（国内游/出境游）

**核心痛点**：
- 热门景点（故宫、天安门、国博）门票预约难、放票规则复杂
- 捡漏机会稍纵即逝
- 不同平台预约规则复杂，容易遗漏
- 临时变动（天气、政策）影响预约策略

## 2. 核心难点讨论与对策

### 2.1 消息触达硬伤
**问题**：微信小程序订阅消息存在"发一次消耗一个授权"的限制

**对策**：采用组合提醒策略
- 云函数生成 .ics 手机日历同步文件（提供持久化闹钟提醒）
- 微信订阅消息（放票前即时通知）
- 公众号/企微机器人联动（突破订阅限制）
- 微信群机器人@所有人（100%触达率）

### 2.2 数据源获取（护城河）
**问题**：热门景区数据多闭环在小程序内，网页版数据易被复制

**对策**：
- V1阶段：优先抓取有官网网页版的景区（天安门城楼等）
- V2阶段：通过RPA或协议分析攻克小程序闭环数据
- 众包反馈机制：用户发现数据不准可一键报错

### 2.3 反爬策略对抗
**问题**：官方渠道有IP封禁、WebDriver检测等反爬手段

**对策**：
- 使用Playwright-stealth插件抹除自动化标记
- 接入动态住宅代理（快代理等）
- 随机生成浏览器指纹和User-Agent
- 分时段监控：非高峰期低频，放票前高频

### 2.4 微信生态限制
**问题**：小程序跳转限制、推送配额限制

**对策**：
- 官方小程序直跳（wx.navigateToMiniProgram）
- 无小程序则提供复制链接+操作截图
- 订阅合集：引导用户授权多种通知类型
- 企业微信机器人实现无限次推送

## 3. 产品版本规划

### V1版本（当前目标）：信息聚合与基础提醒
**核心功能**：
- 热门景点实现查看最早可定日期
- 支持用户按目标出行日期设置最早可预订时间提醒（公众号消息和手机日历）
- 景点预约入口汇总（类似携程，按直达率排序）
- 我的预约（取消收藏功能）

**技术重点**：
- 跑通5个核心Web景区抓取 + 微信直跳 + 搜索词优化
- 数据模型：Spots（景点表）、Rules（规则表）、Subscriptions（订阅表）
- 核心指标：日活用户数(DAU)

### V2版本：余票监控内测
**核心功能**：
- 引入RPA或API抓取小程序闭环数据
- 引入公众号/群机器人通知
- 智能备选方案逻辑（当目标景点无票时推荐附近替代）

**技术重点**：
- 攻克小程序内数据抓取技术
- 核心指标：补票成功率

### V3版本：商业化与AI增强
**核心功能**：
- 增加"智能周边推荐" + AI旅行路线规划 + 景区热力图查询
- 付费/会员制服务（高级提醒策略、专属客服）
- 分级收费：常规提醒免费，瞬时捡漏提醒付费

**技术重点**：
- AI旅行规划功能
- 核心指标：留存率/转化率

## 4. V1版本核心技术方案

### 4.1 技术架构
```
前端（小程序端）：微信原生开发 + Vant Weapp
    ↓
后端（云开发）：腾讯云云开发 (Cloud Development)
    ↓
抓取层：部署在云函数上的 Playwright + stealth插件
    ↓
数据存储：云数据库 (NoSQL)
```

### 4.2 核心数据模型 (Data Schema)

#### Spots (景点表)
```javascript
{
  spotId: String,          // 唯一标识符
  name: String,           // 景点名称
  category: String,       // 景点分类（博物馆、公园、古迹）
  officialAppid: String,  // 官方购票小程序AppID
  officialPath: String,   // 官方购票小程序路径（Deep Link）
  officialWebUrl: String, // 网页版预约地址（备份）
  location: Object,       // 经纬度坐标
  scrapingSelector: String, // 爬虫CSS选择器
  difficultyScore: Number, // 预约难度指数
  popularityScore: Number  // 热度指数
}
```

#### Rules (规则表)
```javascript
{
  spotId: String,        // 关联景点ID
  advanceDays: Number,   // 提前放票天数（如：7天）
  releaseTime: String,   // 每日精准放票时间（如：20:00:00）
  isRolling: Boolean,    // 是否滚动放票
  bookingTips: String,   // 预约注意事项
  lastCheckedDate: Date  // 最新可约日期
}
```

#### Subscriptions (订阅表)
```javascript
{
  userId: String,        // 用户OpenID
  spotId: String,        // 订阅的景点ID
  targetDate: Date,      // 用户计划出行的日期
  notifyStatus: Enum,    // 提醒状态（待提醒、已过期、已推送）
  channels: Array       // 订阅渠道（日历、消息、群机器人）
}
```

### 4.3 核心功能实现

#### 4.3.1 数据自动化抓取 (Scraper)
- **初期路径**：针对故宫、天安门城楼等有Web官网的景点编写爬虫脚本
- **逻辑**：每日0点及各放票点后5分钟，自动访问官网，解析日历HTML元素，提取最新"可预约"日期
- **异常处理**：抓取失败时云函数自动给开发者发送告警

#### 4.3.2 多维度提醒逻辑 (Notifier)
- **手机日历同步**：云函数生成.ics文件链接，日历备注包含跳转链接
- **订阅消息**：放票前5分钟触发一次性订阅消息
- **分流提醒**：根据用户订阅顺序在30秒内错峰推送，避免官方接口拥堵

#### 4.3.3 入口汇总与跳转 (The Hub)
- **UI交互**：类似携程的聚合面板，按"直达率"排序
- **跳转实现**：
```javascript
wx.navigateToMiniProgram({
  appId: spot.officialAppid,
  path: spot.officialPath,
  success(res) { /* 成功跳转 */ }
})
```
- **兜底方案**：无小程序则提供"复制链接"+操作截图提示

#### 4.3.4 智能备选方案逻辑
当用户设置的目标日期无票时：
- 基于地理位置推荐附近替代景点
- 示例："故宫已约满。为您发现距离2km的【景山公园】目前仍有余票，可俯瞰故宫全景"

## 5. 商业模式与运营策略

### 5.1 盈利模式
- **前期**：纯免费"流量型工具"，靠口碑和社交裂变获客
- **后期**：接入广告或旅游产品带货
- **远期**：付费/会员制服务（更及时、更高频率的捡漏监控）

### 5.2 冷启动策略
- **SEO截流**：每个景点名称作为流量入口，做"XX景点预约规则说明"页面
- **小红书引流**：利用设计师优势制作精美攻略图文
- **提醒分享**："帮好友代设提醒"功能，实现社交裂变
- **家庭账户**：一人设提醒，全家同步收到通知

### 5.3 成本预估（月度）
- **起步期**：约230元（服务器50元 + 代理IP 150元 + 其他30元）
- **增长期**：约1710元（用户量过万后）

### 5.4 收入预测（6个月）
- **5月（五一）**：3,000-5,000元
- **7-8月（暑假）**：15,000-40,000元/月
- **10月（国庆）**：30,000-50,000元
- **6个月累计**：约80,000-132,000元

## 6. 竞争格局分析

### 6.1 三类竞争者
1. **大平台"全家桶"**（携程、飞猪、美团）：流量大但响应不及时
2. **官方聚合平台**（京通小程序、畅游公园）：数据准确但功能死板
3. **垂直"游击队"**（游抢抢、博物馆预约助手）：功能垂直但UI粗糙、稳定性差

### 6.2 反超点
- **"设计师级别"的UX**：利用专业UI优势做出差异化
- **"备选方案"逻辑**：从"监控器"变成"咨询师"
- **"家庭组"提醒**：针对亲子游场景的差异化功能

## 7. 关键执行建议

### 7.1 给Claude Code的执行指令
- **特征隐藏**：编写抓取脚本时必须包含stealth.min.js，设置真实User-Agent
- **分流策略**：设计提醒发送逻辑时预留"分流提醒"接口
- **众包机制**：前端预留"一键报错"按钮用于数据准确性反馈

### 7.2 实施优先级
1. **MVP阶段**：先跑通天安门城楼等1-2个核心景点
2. **护城河建设**：攻克小程序数据抓取技术
3. **商业化验证**：通过广告和会员制验证商业模式

### 7.3 风险控制
- **法律红线**：严禁涉及"代抢代约"并收费
- **技术风险**：建立监控心跳系统，秒级感知数据源异常
- **投诉风险**：显著位置声明"本工具仅提供提醒服务，不保证100%抢票成功"
